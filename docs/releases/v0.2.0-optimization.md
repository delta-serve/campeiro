# [v0.2.0] Otimização de Performance e Observabilidade

## 📅 Data: 27 de fevereiro de 2026

## 🎯 Objetivo
Otimizar performance do pipeline através da redução de logs verbosos (-90% de volume) e melhorar observabilidade com métricas críticas adicionais e dashboards Grafana robustos.

## 📋 Descrição

### Contexto de Negócio
Após a implantação inicial (v0.1.0), identificamos dois problemas operacionais:
1. **Volume excessivo de logs:** ~10GB/dia de logs DEBUG, dificultando troubleshooting
2. **Métricas insuficientes:** Falta de visibilidade granular sobre erros e latências por destino

Esta versão foca em tornar o pipeline **production-ready** através de otimizações de observabilidade.

### Decisões Técnicas Principais

#### 1. Redução de Logs Verbosos (-90%)
**Problema identificado:** Logs em nível DEBUG geravam 10GB/dia, saturando armazenamento.

**Solução implementada:**
- Alteração de nível `DEBUG` → `INFO` para operações normais
- Logs `DEBUG` mantidos apenas para erros e retries
- Structured logging em JSON para parsing eficiente
- Rotação de logs configurada (1GB por arquivo, retenção de 7 dias)

**Resultado:** Redução de 10GB/dia → 1GB/dia (-90%)

#### 2. Métricas Críticas Adicionadas
Além das 6 métricas base, foram adicionadas:

```yaml
metrics:
  prometheus:
    # Métricas existentes (v0.1.0)
    - input_received_total
    - input_latency_ns
    - output_sent_total
    - output_error_total
    - processor_error_total
    - benthos_running
    
    # NOVAS métricas críticas (v0.2.0)
    - output_sent_by_destination{destination}
    - output_error_by_destination{destination}
    - output_latency_by_destination_seconds{destination}
    - pipeline_throughput_events_per_second
    - kafka_consumer_lag_seconds
```

**Justificativa:** Permite identificar destinos problemáticos e gargalos específicos.

#### 3. Dashboards Grafana Robustos
**Dashboard 1: Overview Executivo**
- Total de eventos processados (24h)
- Taxa de sucesso global
- Latência P95 agregada
- Alertas ativos

**Dashboard 2: Análise por Destino**
- Throughput por destino (Mauá, Osasco, SIMTRANS)
- Taxa de erro por destino (%)
- Latência P50/P95/P99 por destino
- Consumer lag por tópico

**Dashboard 3: Troubleshooting**
- Eventos em retry
- Erros agrupados por tipo
- Timeline de incidentes
- Correlação entre latência e erro

**Queries otimizadas:** Uso de `rate()` e `histogram_quantile()` para performance em janelas de 24h+

#### 4. Alertas Configurados (Prometheus Alertmanager)
```yaml
alerts:
  - alert: HighErrorRate
    expr: rate(output_error_total[5m]) > 10
    severity: warning
    
  - alert: HighLatency
    expr: histogram_quantile(0.95, output_latency_seconds) > 1.0
    severity: warning
    
  - alert: ConsumerLagHigh
    expr: kafka_consumer_lag_seconds > 300
    severity: critical
```

```yaml

    pipeline: ab-retorno-renavam
    version: v0.2.0
```

#### Métricas com Labels
```yaml
metrics:
  prometheus:
    prefix: bento_ab_retorno
    push_interval: 10s
    push_job_name: ab-retorno-renavam
    
  mapping: |
    meta destination = this.output.label
    meta status = if this.error == null { "success" } else { "error" }
```

### Impacto
- **Logs:** Redução de 90% no volume (10GB → 1GB/dia)
- **Observabilidade:** +5 métricas críticas com labels
- **Dashboards:** 3 dashboards Grafana operacionais
- **Alertas:** 3 alertas configurados (erro, latência, lag)
- **Performance:** Mesma taxa de throughput (~500 eventos/min) com menor overhead

### Aprendizados
1. **Logs estruturados:** JSON facilita parsing e análise automatizada
2. **Métricas com labels:** Essencial para troubleshooting em sistemas multi-destino
3. **Dashboards por persona:** Executivo, Operador e SRE têm necessidades distintas
