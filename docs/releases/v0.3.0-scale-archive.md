# [v0.3.0] Escala Multi-Destino e Arquivamento Raw

## 📅 Data: 9 de março de 2026

## 🎯 Objetivo
Expandir pipeline para 4º destino (SSP-NAVEGANTES/SC) e implementar arquivamento de eventos raw para auditoria e replay, mantendo performance e confiabilidade.

## 📋 Descrição

### Contexto de Negócio
A SSP (Secretaria de Segurança Pública) de Navegantes/SC foi incluída no projeto Alerta Brasil e necessita receber eventos de retorno RENAVAM. Adicionalmente, surgiu a necessidade de:
1. **Arquivar eventos originais** para auditoria e conformidade legal
2. **Capacidade de replay** em caso de falhas ou reprocessamento

### Decisões Técnicas Principais

#### 1. Novo Destino: SSP-NAVEGANTES/SC
**Perfil do parceiro:**
- **Empresa:** PROTENET (equipamentos de fiscalização)
- **Volume esperado:** 200-300 eventos/dia
- **Protocolo:** Kafka (HTTP será implementado em v1.0.0)
- **Tópico:** `navegantes-sc.eventos.renavam`

**Configuração:**
```yaml
- label: kafka_navegantes_sc
  kafka:
    addresses: ["${KAFKA_BROKERS}"]
    topic: navegantes-sc.eventos.renavam
    partitions: 3
    compression: gzip
    max_message_bytes: 1048576
```

#### 2. Arquivamento de Eventos Raw
**Motivação:**
- Conformidade com LGPD (auditoria de 5 anos)
- Capacidade de replay para correção de erros
- Análise forense de incidentes

**Implementação: Padrão Fan-Out**
```yaml
output:
  broker:
    pattern: fan_out
    outputs:
      # 4 destinos operacionais
      - label: kafka_maua_sp
      - label: kafka_osasco_sp
      - label: kafka_simtrans_vca_ba
      - label: kafka_navegantes_sc
      
      # Arquivamento raw (novo)
      - label: kafka_archive_raw
        kafka:
          topic: alerta-brasil.archive.renavam.raw
          compression: snappy
          retention_days: 1825  # 5 anos (LGPD)
```

**Formato de arquivamento:**
- Eventos originais **sem transformação**
- Compressão Snappy (melhor custo/benefício)
- Particionamento por data (facilita consultas)
- Retenção de 5 anos (1825 dias)

#### 3. Fan-Out Pattern para Múltiplos Consumidores
**Antes (v0.2.0):** Pipeline enviava para 3 destinos
**Agora (v0.3.0):** Pipeline envia para 5 destinos (4 operacionais + 1 arquivo)

**Vantagens do fan-out:**
- Processamento paralelo e independente
- Falha em um destino não afeta os demais
- Facilita adição de novos destinos

**Configuração de paralelismo:**
```yaml
output:
  broker:
    pattern: fan_out
    max_in_flight: 64  # Permite até 64 mensagens em paralelo
    batching:
      count: 100
      period: 1s
```

#### 4. Otimização de Performance
Com 5 destinos simultâneos, foram necessários ajustes:

**Buffer de memória aumentado:**
```yaml
buffer:
  memory:
    limit: 256MB  # Antes: 128MB
    batch_policy:
      enabled: true
      count: 100
```

**Compressão Kafka:**
- **Destinos operacionais:** GZIP (melhor compressão)
- **Arquivo raw:** Snappy (melhor velocidade)

**Timeouts ajustados:**
- Write timeout: 30s (antes: 10s)
- Connection timeout: 10s (antes: 5s)
