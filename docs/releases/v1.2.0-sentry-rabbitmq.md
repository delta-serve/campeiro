# [Ativação] Envio de Dados para Empresa SENTRY via RabbitMQ

## 🎯 Objetivo
Ativar envio de eventos de retorno RENAVAM para a empresa SENTRY via RabbitMQ, consolidando o atendimento aos órgãos gerenciados por ela (MAUA-SP e OSASCO-SP).

## 📋 Descrição

### Contexto de Negócio
A SENTRY é uma empresa de gerenciamento de equipamentos de fiscalização eletrônica que opera nos municípios de **Mauá/SP** e **Osasco/SP**. Similar ao modelo implementado com a PROTENET (Navegantes/SC), os eventos de retorno RENAVAM serão roteados por empresa e enviados via RabbitMQ para um endpoint centralizado da SENTRY.

**Objetivo de Negócio:** Consolidar o envio de eventos para órgãos gerenciados pela mesma empresa, otimizando a distribuição e simplificando a integração.

### Decisões Técnicas Principais

#### 1. Modelo de Roteamento por Empresa
**Regra de roteamento:**
```bloblang
root = match {
  this.empresa == "PROTENET" && this.equipamento.starts_with("RADAR") => "http_navegantes_sc",
  this.empresa == "SENTRY" => "rabbitmq_sentry",  # NOVA REGRA
  this.empresa == "SIMTRANS_VCA" => "kafka_simtrans_vca_ba",
  _ => "dlq_unrouted"
}
```

**Órgãos atendidos pela SENTRY:**
- Prefeitura de Mauá/SP (MUNICIPAL_MAUA)
- Prefeitura de Osasco/SP (MUNICIPAL_OSASCO)

#### 2. Integração RabbitMQ
**Configuração base (aguardando detalhes):**

```yaml
output:
  broker:
    pattern: fan_out
    outputs:
      # ... outputs existentes ...
      
      - label: rabbitmq_sentry
        amqp_1:  # ou amqp_0_9 (aguardando definição)
          urls:
            - "${RABBITMQ_SENTRY_URL}"
          exchange: "${RABBITMQ_SENTRY_EXCHANGE}"
          exchange_type: "${RABBITMQ_SENTRY_EXCHANGE_TYPE}"
          key: "${RABBITMQ_SENTRY_ROUTING_KEY}"
          metadata:
            exclude_prefixes: []
          max_in_flight: 64
```

**Variáveis de ambiente necessárias (.env):**
```bash
# RabbitMQ SENTRY Configuration
RABBITMQ_SENTRY_URL=amqp://user:password@host:port/vhost
RABBITMQ_SENTRY_EXCHANGE=sentry.eventos.renavam
RABBITMQ_SENTRY_EXCHANGE_TYPE=topic  # ou direct/fanout
RABBITMQ_SENTRY_ROUTING_KEY=alerta-brasil.retorno.renavam
RABBITMQ_SENTRY_VHOST=/sentry  # se aplicável
```

#### 3. Resiliência e Retry
**Configuração de retry para RabbitMQ:**
```yaml
      - label: rabbitmq_sentry
        processors:
          - retry:
              max_retries: 3
              backoff:
                initial_interval: 1s
                max_interval: 60s
                max_elapsed_time: 5m
```

#### 4. Métricas Adicionais
**Novas métricas para monitoramento:**
```yaml
metrics:
  prometheus:
    - rabbitmq_messages_sent_total{destination="sentry"}
    - rabbitmq_messages_error_total{destination="sentry"}
    - rabbitmq_connection_status{destination="sentry"}
    - rabbitmq_publish_latency_seconds{destination="sentry"}
```

### Impacto Esperado

#### Consolidação de Destinos
**Antes:**
- 2 destinos Kafka: kafka_maua_sp, kafka_osasco_sp
- 1 destino HTTP: http_navegantes_sc (PROTENET)
- 1 destino Kafka: kafka_simtrans_vca_ba

**Depois:**
- 1 destino RabbitMQ: rabbitmq_sentry (atende MAUA + OSASCO)
- 1 destino HTTP: http_navegantes_sc (PROTENET)
- 1 destino Kafka: kafka_simtrans_vca_ba

**Total de empresas atendidas:** 3 (SENTRY, PROTENET, SIMTRANS)

#### Performance
- **Throughput esperado:** ~1.500 eventos/min (incluindo MAUA e OSASCO)
- **Latência P95:** < 200ms
- **Taxa de sucesso:** 99.9% (com retry)
