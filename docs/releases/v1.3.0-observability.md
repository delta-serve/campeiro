# [v1.3.0] Health Check Logs Periódicos - Observabilidade Proativa

## 🎯 Objetivo
Adicionar logs de health check periódicos ao pipeline AB Retorno RENAVAM para facilitar monitoramento proativo e troubleshooting sem necessidade de polling constante de métricas Prometheus/Grafana.

## 📋 Descrição

### Contexto de Negócio
Após a implementação bem-sucedida da integração RabbitMQ SENTRY (v1.2.0), identificou-se a necessidade de melhorar a observabilidade operacional do pipeline. Durante análise de logs em produção, ficou evidente que:

> "Não temos erros, pois estou filtrando quaisquer mensagens e não tem nada após. O interessante seria se tivéssemos uma mensagem no log a cada X minutos informando que está tudo bem."

**Objetivo Operacional:** Facilitar o troubleshooting proativo, permitindo que operadores identifiquem rapidamente o estado do pipeline sem necessidade de consultar dashboards ou executar comandos manuais.

### Decisões Técnicas Principais

#### 1. Implementação com Branch Processor

**Estratégia escolhida:** Branch processor não-bloqueante

**Vantagens:**
- ✅ Zero impacto no fluxo de mensagens (`result_map: deleted()`)
- ✅ Não adiciona latência ao processamento
- ✅ Não afeta throughput
- ✅ Isolado do pipeline principal (falha no health check não para o pipeline)

**Configuração:**
```yaml
- branch:
    request_map: |
      # Condição temporal (a cada 15 min) + amostragem aleatória (1/8)
      # Reduz logs duplicados quando múltiplas mensagens processadas no mesmo segundo
      root = if now().ts_unix() % 900 == 0 && random_int(min: 0, max: 7) == 0 { 
        this 
      } else { 
        deleted() 
      }
    processors:
      - log:
          level: INFO
          fields_mapping: |
            root.health_check = { ... }
    result_map: 'root = deleted()'
```

#### 2. Controle de Frequência Temporal

**Mecanismo:** Condição baseada em `timestamp_unix() % 900` + amostragem aleatória

**Frequência:** ~15 minutos (900 segundos)

**Funcionamento:**
- A cada mensagem processada, verifica se `now().ts_unix() % 900 == 0`
- Adicionalmente aplica `random_int(min: 0, max: 7) == 0` (probabilidade 1/8)
- Isso reduz logs duplicados quando múltiplas mensagens processadas no mesmo segundo
- Resultado: ~1 log a cada 15 minutos (em vez de 1 log por mensagem processada naquele segundo)

**Alternativas consideradas e descartadas:**
- ❌ `rate_limit` com resource: Requer seção `resources` (não suportada no Bento 1.10.0)
- ❌ `sample` com retain: Não disponível como processor no Bento 1.10.0
- ❌ `throttle` processor: Não existe no Bento
- ✅ **Bloblang temporal + amostragem (escolhida):** Simples, confiável, sem dependências

#### 3. Formato do Log

**Dual-format:** JSON estruturado + mensagem legível

**JSON (para parsing automatizado):**
```json
{
  "health_check": {
    "pipeline": "ab_retorno_renavam",
    "timestamp": "2026-03-13 09:15:00",
    "timestamp_unix": 1741867500,
    "outputs": {
      "total": 4,
      "active": ["eventos_kafka", "sentry_rabbitmq", "protenet_http", "simtrans_kafka"]
    },
    "empresas_suportadas": ["SENTRY (MULTIWAY)", "PROTENET"],
    "protocolos": ["Kafka", "AMQP 0.9", "HTTP"],
    "status": "HEALTHY"
  }
}
```

**Mensagem legível (para visualização rápida):**
```
========================================
[HEALTH CHECK] Pipeline AB Retorno RENAVAM
Status: OPERACIONAL ✓
Outputs Ativos: 4/4
Empresas: 2 (SENTRY, PROTENET)
Protocolos: 3 (Kafka, AMQP, HTTP)
========================================
```

#### 4. Informações Incluídas no Health Check

| Campo | Tipo | Descrição |
|-------|------|-----------|
| `pipeline` | String | Nome do pipeline (`ab_retorno_renavam`) |
| `timestamp` | String | Data/hora formatada (fuso São Paulo) |
| `timestamp_unix` | Integer | Unix timestamp (para cálculos) |
| `outputs.total` | Integer | Total de outputs configurados (4) |
| `outputs.active` | Array | Lista de outputs ativos |
| `outputs.destinations` | Object | Mapa output → descrição |
| `empresas_suportadas` | Array | Empresas roteadas (SENTRY, PROTENET) |
| `protocolos` | Array | Protocolos em uso (Kafka, AMQP, HTTP) |
| `status` | String | Estado geral (`HEALTHY`) |

## 🔧 Configuração

### Não Requer Variáveis de Ambiente

Health check usa apenas funções built-in do Bloblang:
- `now()` - Timestamp atual
- `ts_unix()` - Conversão para Unix timestamp
- `format_timestamp()` - Formatação de data/hora

**Vantagem:** Funciona out-of-the-box, sem configuração adicional.

## 🔄 Update/Hotfix: Otimização de Logs Duplicados

**Data:** 13 Mar 2026, 11:15  
**Commit:** [a ser gerado]

### Problema Identificado

Durante validação em produção (13 Mar 2026, 11:00:00), observou-se:
- ✅ Health check funcionando corretamente
- ⚠️ **7 logs idênticos no mesmo segundo**
- Causa: Múltiplas mensagens processadas no segundo exato quando `timestamp_unix % 900 == 0`

**Exemplo dos logs duplicados:**
```bash
Mar 13 11:00:00 svn0145 bento[2722920]: [HEALTH CHECK] Pipeline AB Retorno RENAVAM
Mar 13 11:00:00 svn0145 bento[2722920]: [HEALTH CHECK] Pipeline AB Retorno RENAVAM
Mar 13 11:00:00 svn0145 bento[2722920]: [HEALTH CHECK] Pipeline AB Retorno RENAVAM
# ... 4 linhas adicionais idênticas
```

### Solução Implementada

**Amostragem Adicional:** Adicionar condição `random_int()` para reduzir probabilidade de múltiplos logs.

**Modificação no código:**
```yaml
# ANTES (commit edfef02):
root = if now().ts_unix() % 900 == 0 { this } else { deleted() }

# DEPOIS (este hotfix):
root = if now().ts_unix() % 900 == 0 && random_int(min: 0, max: 7) == 0 { 
  this 
} else { 
  deleted() 
}
```

**Impacto:**
- Reduz de ~7 logs para ~1 log a cada 15 minutos (probabilidade 1/8 = 12.5%)
- Mantém funcionalidade: health check continua aparecendo periodicamente
- Zero impacto no fluxo de mensagens (branch não-bloqueante)

**Arquivos modificados:**
1. `streams/ab_retorno_renavam.yaml` (linha ~149)
2. `docs/OBSERVABILITY.md` (seção "Ajustar Frequência")
3. `docs/releases/v1.3.0-observability.md` (esta seção)

## 🎓 Lições Aprendidas

### 1. Limitações do Bento 1.10.0

**Descoberta:** Seção `resources` não é suportada na versão 1.10.0

**Impacto:** Não foi possível usar `rate_limit` processor com resource compartilhado

**Solução:** Usar condição temporal em Bloblang (`timestamp_unix() % 900`)

**Aprendizado:** Sempre verificar documentação da versão específica antes de implementar

### 2. Branch Processor para Side Effects

**Padrão descoberto:** Branch com `result_map: deleted()` é ideal para side effects

**Benefícios:**
- Permite executar ações (logs, métricas) sem afetar fluxo principal
- Zero risco de corromper dados
- Isolamento de falhas

**Aplicabilidade:** Pode ser usado para outras funcionalidades futuras (auditing, sampling, debug)

### 3. Health Checks Complementam Métricas

**Insight:** Logs qualitativos + métricas quantitativas = observabilidade completa

**Health Checks são melhores para:**
- Status geral (HEALTHY/DEGRADED)
- Configuração atual (outputs, empresas)
- Troubleshooting rápido (grep nos logs)

**Métricas são melhores para:**
- Análise temporal (tendências)
- Alertas automáticos (thresholds)
- Dashboards detalhados

**Conclusão:** Ambos são necessários e complementares
