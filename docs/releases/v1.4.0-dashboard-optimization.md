# [v1.4.0] Otimização Dashboard Grafana - Redução de Complexidade Visual

## 🎯 Objetivo
Otimizar o dashboard Grafana do pipeline AB Retorno RENAVAM, reduzindo complexidade visual de 43 para 29 painéis através de hierarquia clara com rows colapsadas, priorizando KPIs críticos.

## 📋 Descrição

### Contexto de Negócio

O dashboard atual apresentava 43 painéis todos visíveis simultaneamente, causando:
- ❌ **Poluição visual:** Fadiga cognitiva para operadores
- ❌ **Dificuldade em identificar problemas:** KPIs críticos dispersos
- ❌ **Falta de hierarquia:** Tudo com mesma importância visual
- ❌ **Redundância:** HTTP e RabbitMQ com estrutura idêntica (12 painéis duplicados)
- ❌ **Scroll excessivo:** Operadores precisavam rolar muito para ver erros/latências

**Problema identificado:** Durante troubleshooting, operadores gastavam ~30 segundos rolando e buscando informações críticas.

**Objetivo:** Reduzir tempo de detecção de problemas de 30s para ~3s através de visão consolidada no topo.

## 🏗️ Decisões Técnicas Principais

### 1. Estrutura Hierárquica com Rows Colapsadas

**Estratégia escolhida:** 5 rows organizadas por prioridade operacional

```
ROW 1: OVERVIEW (visível) - KPIs críticos e consolidados
ROW 2: DISTRIBUIÇÃO NEGÓCIO (visível) - Status, Órgão, Empresa, Equipamentos
ROW 3: DETALHAMENTO OUTPUTS (colapsada) - Kafka, HTTP, RabbitMQ lado a lado
ROW 4: ANÁLISE VEÍCULOS (colapsada) - Dados de negócio
ROW 5: KAFKA ARCHIVE (colapsada) - Eventos brutos
```

**Vantagens:**
- ✅ Visão inicial limpa (11 painéis vs 43 anteriormente)
- ✅ Informação crítica acessível sem scroll
- ✅ Detalhes técnicos disponíveis em 1 clique (rows expansíveis)
- ✅ Redução 74% de scroll necessário

### 2. Novos Painéis Críticos

#### a) Taxa de Erro % (NOVO)

**Query Prometheus:**
```promql
(sum(rate(output_error_total{label=~"ab_retorno_renavam.*"}[5m])) 
 / 
 sum(rate(renavam_messages_received_total[5m]))) * 100
```

**Thresholds:**
- Verde: 0-0.1%
- Amarelo: 0.1-1%
- Vermelho: > 1%

**Justificativa:** Métrica mais importante que contagem absoluta de erros. Permite identificar degradação proporcionalmente ao volume.

---

#### b) Health Status (NOVO)

**Tipo:** Gauge (0-100%)

**Fórmula:**
```
Health = 100 
         - (erro% * 10)              // Penaliza 10 pontos por 1% de erro
         - (latência_p95 > 1s ? 20 : 0)  // Penaliza 20 pontos se latência alta
```

**Query Prometheus:**
```promql
100 
- clamp_min(
    (sum(rate(output_error_total{label=~"ab_retorno_renavam.*"}[5m])) 
     / sum(rate(renavam_messages_received_total[5m]))) * 100 * 10, 
    0
  )
- clamp_min(
    (clamp_max(
      histogram_quantile(0.95, sum(rate(output_latency_ns_bucket[5m])) by (le)) / 1e6, 
      10000
    ) > 1000) * 20, 
    0
  )
```

**Thresholds:**
- Verde (90-100%): Sistema saudável
- Amarelo (70-90%): Atenção necessária
- Vermelho (< 70%): Problema crítico

**Vantagens:**
- ✅ Indicador único de saúde do sistema
- ✅ Combina erros + latência em métrica visual
- ✅ Gauge permite identificação imediata (verde/amarelo/vermelho)

### 1. Duplicação de Contagem no Painel "Total Output"

**Problema Identificado (commit `ae18a7f`):**

A query original do painel "Total Output" estava somando 3 métricas que contavam as MESMAS mensagens:

```promql
# QUERY INCORRETA (antes da correção)
sum(increase(renavam_messages_sent_total{destination=~"$destination"}[$__range])) 
+ sum(increase(renavam_http_navegantes_sent_total[$__range])) 
+ sum(increase(renavam_rabbitmq_sentry_sent_total[$__range]))
```

**Causa Raiz:**

Análise do arquivo `streams/ab_retorno_renavam.yaml` revelou:

1. **`renavam_messages_sent_total`** (linha 58)
   - Registrada **NO PIPELINE** (antes dos outputs)
   - Conta **TODAS** as mensagens por destination
   - Labels: `destination` (eventos_kafka, sentry, simtrans, navegantes, outros)

2. **`renavam_rabbitmq_sentry_sent_total`** (linha 241)
   - Registrada **DENTRO** do output RabbitMQ
   - Já contada em `renavam_messages_sent_total{destination="sentry"}`

3. **`renavam_http_navegantes_sent_total`** (linha 276)
   - Registrada **DENTRO** do output HTTP
   - Já contada em `renavam_messages_sent_total{destination="navegantes"}`

**Impacto do Bug:**
- Se 1000 mensagens fossem enviadas para SENTRY: mostrava **2000** (duplicado) ❌
- Valores inflados confundiam análise de throughput

**Correção Aplicada:**

```promql
# QUERY CORRETA (após correção)
sum(increase(renavam_messages_sent_total{destination=~"$destination"}[$__range]))
```

**Melhorias Adicionais:**
- ✅ Adicionada **description** explicativa: "Total consolidado de mensagens enviadas para todos os destinos..."
- ✅ Adicionado **legendFormat**: "Total Enviado"
- ✅ Adicionado **unit**: "short" e **decimals**: 0
- ✅ Query agora conta cada mensagem **UMA VEZ APENAS**

**Propósito das Métricas Específicas:**

As métricas `renavam_http_navegantes_sent_total` e `renavam_rabbitmq_sentry_sent_total` são usadas para:
- **Troubleshooting** de problemas específicos de HTTP/RabbitMQ
- **Métricas de latência** (histogram buckets)
- **Taxa de erro por output** (output_error_total)
- **NÃO** para contagem total consolidada

**Lição Aprendida:**
- Métricas em diferentes pontos do pipeline (pipeline vs output) servem propósitos diferentes
- Sempre verificar onde as métricas são registradas antes de somá-las
- Adicionar descriptions em painéis complexos previne reintrodução de bugs

### 2. Duplicação de Contagem no Painel "Taxa Consolidada por Protocolo"

**Problema Identificado (commit `91fc38b`):**

Após análise completa do dashboard, identificou-se um **segundo painel** com o mesmo padrão de duplicação.

**Painel afetado:**
- Nome: "Taxa Consolidada por Protocolo (msg/s)"
- Tipo: timeseries (gráfico de linhas)
- Localização: Row 1 (Overview) - painel grande (h=8, w=16)
- Queries: A (Kafka), B (HTTP), C (RabbitMQ), D (Total)

**Query D antes da correção (INCORRETA):**
```promql
sum(rate(renavam_messages_sent_total{destination=~"$destination"}[1m])) 
+ sum(rate(renavam_http_navegantes_sent_total[1m])) 
+ sum(rate(renavam_rabbitmq_sentry_sent_total[1m]))
```

**Impacto do Bug:**
- Linha "Total" (verde tracejada) mostrava valores inflados
- Se 100 msg/s Kafka + 50 msg/s HTTP + 30 msg/s RabbitMQ
- Total mostrava: 100 + 50 + 30 + 50 + 30 = **260 msg/s** ❌ (duplicação)
- Valor correto seria: **180 msg/s** ✅

**Correção Aplicada:**

```promql
# Query D corrigida (conta cada mensagem uma vez)
sum(rate(renavam_messages_sent_total{destination=~"$destination"}[1m]))
```

**Melhorias Adicionais:**
- ✅ Query D simplificada (remove duplicação)
- ✅ Adicionada **description**: "Taxa de envio por protocolo (msg/s). Queries A/B/C mostram protocolos separados..."
- ✅ Queries A/B/C mantidas (mostram separação visual Kafka/HTTP/RabbitMQ)
- ✅ Linha "Total" verde tracejada agora mostra taxa real
